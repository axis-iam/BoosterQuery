Guide使用指南

Learn how to use BoosterQuery to enhance your Spring Data JPA native SQL experience. 了解如何使用 BoosterQuery 增强 Spring Data JPA 原生 SQL 开发体验。

Installation安装

BoosterQuery requires Java 21+ and Spring Boot 4.0+. BoosterQuery 需要 Java 21+ 和 Spring Boot 4.0+。 

implementation("com.chaosguide:booster-query:1.0.0")
<dependency>
    <groupId>com.chaosguide</groupId>
    <artifactId>booster-query</artifactId>
    <version>1.0.0</version>
</dependency>

Quick Start快速开始

Enable BoosterQuery by setting the repository factory bean class: 通过设置 Repository 工厂 Bean 类启用 BoosterQuery： 

@SpringBootApplication
@EnableJpaRepositories(repositoryFactoryBeanClass = BoosterQueryRepositoryFactoryBean.class)
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}

Define your repository by extending BoosterQueryRepository: 定义 Repository，继承 BoosterQueryRepository： 

public interface UserRepository extends BoosterQueryRepository<User, Long> { }

Now you can execute native SQL with enhanced features: 现在可以使用智能特性执行原生 SQL： 

@Service
public class UserService {
    private final UserRepository repo;

    public List<User> search(String name, Integer age) {
        String sql = "SELECT * FROM t_user WHERE name LIKE :name AND age > :age";
        Map<String, Object> params = new HashMap<>();
        params.put("name", name);  // null → condition removed automatically
        params.put("age", age);    // null → condition removed automatically
        return repo.boosterQueryList(sql, params);
    }
}

Repository System仓库体系

BoosterQuery provides a two-tier repository system: BoosterQuery 提供两层仓库体系：

Interface接口 Methods Prefix方法前缀 Features特性
BoosterNativeRepository native* Basic native SQL execution + result mapping基础原生 SQL 执行 + 结果映射
BoosterQueryRepository booster* All basic features + SQL rewriting + auto-limit + caching全部基础特性 + SQL 改写 + 自动限制 + 缓存
Tip BoosterQueryRepository extends BoosterNativeRepository, which extends JpaRepository. You get all standard Spring Data JPA methods plus the enhanced ones. BoosterQueryRepository 继承 BoosterNativeRepository，后者继承 JpaRepository。你可以同时使用所有标准 Spring Data JPA 方法和增强方法。

Parameter Binding参数绑定

BoosterQuery supports three parameter binding styles: BoosterQuery 支持三种参数绑定方式：

1. Map Parameters1. Map 参数

Map<String, Object> params = Map.of("name", "Alice", "age", 18);
repo.boosterQueryList("SELECT * FROM t_user WHERE name = :name AND age > :age", params);

2. POJO Parameters2. POJO 参数

record UserQuery(String name, Integer age) {}
repo.boosterQueryList("SELECT * FROM t_user WHERE name = :name AND age > :age", new UserQuery("Alice", 18));

3. No Parameters3. 无参数

repo.boosterQueryList("SELECT * FROM t_user ORDER BY id");
Tip POJO binding uses reflection to extract fields. It supports inheritance and works with Java records. Only parameters that exist in the SQL are bound — extra fields are safely ignored. POJO 绑定通过反射提取字段，支持继承和 Java record。只有 SQL 中存在的参数才会被绑定，多余字段会被安全忽略。

4. @BoosterQuery Annotation Parameters4. @BoosterQuery 注解参数

For @BoosterQuery annotated methods, parameter names are resolved in priority order: 对于 @BoosterQuery 注解方法，参数名按以下优先级解析：

  1. @Param annotation — explicit, always works@Param 注解 — 显式声明，始终生效
  2. -parameters compiler flag — infers names via reflection (Spring Boot default)-parameters 编译选项 — 通过反射推断（Spring Boot 默认开启）
  3. Single POJO/Map — extracts fields by name automatically单个 POJO/Map — 自动按字段名提取
// @Param can be omitted when -parameters is enabled (Spring Boot default)
@BoosterQuery("SELECT * FROM t_user WHERE name = :name AND age > :age")
List<User> findByConditions(String name, Integer age);

// @Param is still supported and takes priority
@BoosterQuery("SELECT * FROM t_user WHERE user_name = :userName")
List<User> findByName(@Param("userName") String name);
Note For non-Spring-Boot projects, add -parameters to your compiler options. In Gradle: options.compilerArgs.add("-parameters") 非 Spring Boot 项目需手动添加 -parameters 编译选项。Gradle: options.compilerArgs.add("-parameters")

Result Mapping结果映射

BoosterQuery automatically maps JPA Tuple results to your target type. Underscore column names are converted to camelCase. BoosterQuery 自动将 JPA Tuple 结果映射为目标类型，下划线列名自动转换为驼峰命名。

Supported mapping targets: Entity, DTO (POJO with setters), Record, Interface Projection (via dynamic proxy), Map, and scalar types. 支持的映射目标：Entity、DTO（含 setter 的 POJO）、Record、接口投影（动态代理）、Map 和标量类型。

Entity Mapping (default)Entity 映射（默认）

List<User> users = repo.boosterQueryList("SELECT * FROM t_user");

DTO / Record MappingDTO / Record 映射

record UserDTO(String userName, Integer age) {}
// Column user_name → field userName (auto camelCase)
List<UserDTO> dtos = repo.boosterQueryList(
    "SELECT user_name, age FROM t_user", params, UserDTO.class);

Interface Projection接口投影

Define a getter-only interface and use it as a result type. BoosterQuery creates a dynamic proxy that maps Tuple columns to getter methods (following JavaBean naming conventions). 定义只含 getter 的接口作为结果类型。BoosterQuery 通过动态代理将 Tuple 列映射到 getter 方法（遵循 JavaBean 命名规范）。 

public interface UserProjection {
    String getName();
    String getEmail();
    // default methods are supported
    default String getDisplayName() { return getName() + " <" + getEmail() + ">"; }
}

// Column user_name → getter getUserName (auto camelCase)
List<UserProjection> projections = repo.boosterQueryList(
    "SELECT user_name AS name, email FROM t_user", params, UserProjection.class);
Tip For new code, prefer Record over interface projection — records are immutable, have no proxy overhead, and provide correct equals/hashCode/toString out of the box. 对于新代码，推荐使用 Record 而非接口投影 — Record 不可变、无代理开销，且自动提供正确的 equals/hashCode/toString。

Map MappingMap 映射

List<Map> maps = repo.boosterQueryList(
    "SELECT user_name, age FROM t_user", params, Map.class);

Scalar / Primitive Mapping标量 / 基础类型映射

List<String> names = repo.boosterQueryList(
    "SELECT user_name FROM t_user", params, String.class);

Long count = repo.boosterCount("SELECT * FROM t_user");

Supported scalar types: String, Long, Integer, Double, Float, Short, Byte, BigDecimal, BigInteger, Boolean, Date, LocalDateTime, LocalDate, LocalTime, Timestamp. 支持的标量类型：String、Long、Integer、Double、Float、Short、Byte、BigDecimal、BigInteger、Boolean、Date、LocalDateTime、LocalDate、LocalTime、Timestamp。

SQL RewritingSQL 改写

The SQL rewriter automatically removes conditions from WHERE, HAVING, and JOIN ON clauses when the bound parameter is null, an empty string "", a blank string, or an empty collection. 智能 SQL 改写器在绑定参数为 null、空字符串 ""、空白字符串或空集合时，自动移除 WHERE、HAVING 和 JOIN ON 子句中的对应条件。 

// Original SQL
"SELECT * FROM t_user WHERE name = :name AND age > :age AND status IN :statuses"

// Parameters: {name: "Alice", age: null, statuses: []}
// Rewritten:  SELECT * FROM t_user WHERE name = :name

Structural Expressions结构性表达式

Some SQL patterns intentionally reference parameters in non-filterable contexts, such as :name IS NULL for optional-condition logic or parameters inside CASE WHEN expressions. These are not removed by the rewriter, and the parameters are retained with their value normalized to null. 某些 SQL 模式会在不可过滤的上下文中引用参数，例如 :name IS NULL 用于可选条件逻辑，或 CASE WHEN 表达式中的参数。改写器不会移除这些表达式，参数会被保留且值归一化为 null。 

// Optional-condition pattern: never removed by rewriter
"SELECT * FROM t_user WHERE (:name IS NULL OR name = :name)"

// When name=null → :name IS NULL evaluates to true → returns all rows
// When name="Alice" → name = :name filters normally
Tip Blank strings ("", " ") and empty collections are treated as "null" by the rewriter. When these values appear in structural expressions, they are normalized to actual null to ensure :p IS NULL evaluates correctly. 空字符串（""、" "）和空集合被改写器视为"null"。当这些值出现在结构性表达式中时，会被归一化为实际的 null，确保 :p IS NULL 求值正确。
Safety Check安全拦截 For UPDATE and DELETE statements, if all WHERE conditions are removed (all parameters null), BoosterQuery throws SqlRewriteException to prevent accidental full-table modifications. 对于 UPDATE 和 DELETE 语句，如果所有 WHERE 条件都被移除（所有参数为 null），BoosterQuery 会抛出 SqlRewriteException 以防止意外的全表修改。

Auto-Limit自动限制

Enhanced queries automatically apply setMaxResults to prevent accidentally fetching huge result sets. The default limit is 10,000 rows. 智能查询自动应用 setMaxResults 防止意外获取超大结果集，默认限制 10,000 行。 

// application.yml
booster:
  query:
    default-limit: 5000        # Change default limit
    enable-auto-limit: true   # Enable/disable globally
Tip Auto-limit only applies to booster* methods. The native* methods do not apply auto-limit. You can also override per-method via @BoosterQuery(enableAutoLimit = TRUE, autoLimit = 100). 自动限制仅适用于 booster* 方法，native* 方法不会自动限制。也可以通过 @BoosterQuery(enableAutoLimit = TRUE, autoLimit = 100) 逐方法覆盖。

Caching缓存

BoosterQuery can cache rewritten SQL, count SQL, and sort SQL using Caffeine. Caching is disabled by default. BoosterQuery 可以使用 Caffeine 缓存改写后的 SQL、计数 SQL 和排序 SQL。缓存默认关闭。 

// application.yml
booster:
  query:
    cache:
      enabled: true
      maximum-size: 1000          # Max cached entries
      expire-after-write: 3600000 # TTL in ms (default: 60 min)

@BoosterQuery Annotation@BoosterQuery 注解

Use @BoosterQuery for declarative SQL queries on repository methods, similar to Spring Data's @Query but with enhanced features. 使用 @BoosterQuery 在 Repository 方法上声明式定义 SQL 查询，类似 Spring Data 的 @Query 但具备增强特性。 

public interface UserRepository extends BoosterQueryRepository<User, Long> {

    @BoosterQuery("SELECT * FROM t_user WHERE name LIKE :name AND age > :age")
    List<User> findByConditions(@Param("name") String name, @Param("age") Integer age);

    @BoosterQuery(
        value = "SELECT user_name, email FROM t_user WHERE status = :status",
        resultType = UserDTO.class,
        enableRewrite = Toggle.TRUE,
        enableAutoLimit = Toggle.TRUE,
        autoLimit = 500
    )
    List<UserDTO> findActiveUsers(@Param("status") String status);

    @BoosterQuery(
        value = "SELECT * FROM t_user WHERE dept_id = :deptId",
        countQuery = "SELECT COUNT(*) FROM t_user WHERE dept_id = :deptId"
    )
    Page<User> findByDept(@Param("deptId") Long deptId, Pageable pageable);
}

Supported Return Types支持的返回类型

@BoosterQuery automatically infers the result mapping type from the method's return type: @BoosterQuery 根据方法返回类型自动推断结果映射类型：

Return Type返回类型 Behavior行为
List<T> Query list, maps each row to T列表查询，每行映射为 T
Page<T> Paginated query (requires Pageable parameter)分页查询（需要 Pageable 参数）
Optional<T> Single result wrapped in Optional单条结果包装为 Optional
T (DTO / Record / Interface / Entity) Direct single-result mapping直接单条结果映射
Map<String, Object> Single row as key-value map单行作为键值 Map
String, BigDecimal, etc. Scalar value extraction标量值提取
long / Long Count query计数查询
int / Integer / void DML execution (returns affected rows)DML 执行（返回受影响行数）

Examples — Direct DTO / Record return (no wrapper needed): 示例 — 直接返回 DTO / Record（无需包装类型）： 

// Direct DTO return
@BoosterQuery("SELECT user_name, email FROM t_user WHERE id = :id")
UserDTO findUserById(@Param("id") Long id);

// Direct Record return
@BoosterQuery("SELECT SUM(amount) AS totalRevenue, COUNT(*) AS orderCount FROM t_order")
SummaryRecord getOrderSummary();

// Interface projection return
@BoosterQuery("SELECT name, email FROM t_user WHERE id = :id")
UserProjection findProjectionById(@Param("id") Long id);

// Optional wrapping
@BoosterQuery("SELECT user_name, email FROM t_user WHERE id = :id")
Optional<UserDTO> findOptionalUserById(@Param("id") Long id);
Important注意 Container types (List, Page, Optional) must declare a generic type parameter. Raw types like List (without <T>) will throw IllegalStateException at startup. 容器类型（List、Page、Optional）必须声明泛型参数。使用无泛型的 raw type（如 List）会在启动时抛出 IllegalStateException。
